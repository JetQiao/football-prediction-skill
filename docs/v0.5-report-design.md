# v0.5 离线报告产品与视觉规范

> 状态：已实现并通过静态结构、交互契约与响应式规则校验。
> 方向：可信、克制、高密度的足球概率研究工作台，而不是泛 AI 霓虹展示页。

## 1. 设计目标

报告首先服务三个问题：

1. 今天有哪些比赛值得继续研究？
2. 模型、参考市场和竞彩价格分别怎么看？
3. 哪些比赛应该明确放弃，因为数据不足或没有独立优势？

视觉优先级：

```text
赛单扫描效率 > 概率与价格比较 > 风险/弃权 > 解释细节 > 品牌装饰
```

继续保留：

- 自包含单文件 HTML。
- 离线打开，不发网络请求。
- 打印/PDF。
- 键盘可操作。
- 数据来源、时间和运行 ID 可追溯。

## 2. 现状问题

结合当前模板与真实报告截图：

- Hero 和装饰区过高，第一屏不能快速看到完整赛单。
- 大量 8–10px 字号，桌面端也偏小。
- 英文大写标签过多，信息密度高但有效信息密度不高。
- 绿色同时承担品牌、成功、模型推荐和状态，语义混用。
- 多层卡片、圆角、描边和阴影让页面显得碎。
- 赛单以大卡片为主，场次多时滚动成本高。
- “推荐”比“数据质量”和“弃权原因”更醒目，容易产生过度确定感。
- 概率条能看方向，但无法快速比较模型、参考市场和竞彩 SP。
- 回测图缺少基准线、坐标、置信区间和样本量语境。
- 移动端通过局部横向滚动容纳桌面表格，操作成本高。

## 3. 新的信息架构

### 3.1 桌面首屏

```text
┌──────────────────────────────────────────────────────────────────────┐
│ 竞彩足球研究台  2026-07-16  截点 T-90m  数据 14:32  模型 v0.5.0   │
├──────────────────────────────────────────────────────────────────────┤
│ 18 场  │ 7 场完整数据 │ 2 个候选价值 │ 9 场弃权 │ 校准状态：通过   │
├──────────────────────────────────────────────────────────────────────┤
│ 联赛筛选 │ 时间 │ 数据模式 │ 只看价值 │ 只看异常 │ 搜索球队          │
├────┬───────────────┬────────────┬────────────┬──────────┬────────────┤
│时间│ 对阵          │ 校准概率   │ 公平赔率   │ 竞彩 SP  │ Edge/状态  │
├────┼───────────────┼────────────┼────────────┼──────────┼────────────┤
│19:30 英超 A vs B   │43/29/28    │2.33/3.45… │2.52/3.2…│ +3.1% 候选 │
│20:00 德甲 C vs D   │38/27/35    │2.63/3.70… │2.55/3.4…│ 无优势     │
│20:30 杯赛 E vs F   │—           │—           │1.62/…   │ 数据不足   │
└────┴───────────────┴────────────┴────────────┴──────────┴────────────┘
```

首屏取消大 Hero，改为：

- 一行状态栏。
- 一行关键指标。
- 一行筛选器。
- 立即进入赛单表。

### 3.2 单场详情

点击赛单行后，在当前页面展开详情或打开右侧抽屉：

1. `结论`
   - 校准概率。
   - 公平赔率。
   - 竞彩 SP。
   - 独立 Edge。
   - 置信区间。
   - 弃权/候选价值状态。
2. `概率对照`
   - 动态 Dixon-Coles。
   - 参考市场。
   - 残差模型。
   - 最终校准概率。
3. `比分与进球`
   - 比分热力图。
   - Top 比分。
   - 总进球分布。
   - 让球概率。
4. `赔率轨迹`
   - 开盘、T-24h、T-6h、T-90m 和当前快照。
   - 跨公司分歧。
5. `因素与情报`
   - 结构化特征贡献。
   - 阵容伤停事实。
   - 来源和发布时间。
6. `数据血缘`
   - 每层数据来源、新鲜度和降级原因。
   - 模型、融合器、校准器版本。

### 3.3 回测页

回测报告不复用每日赛单 Hero，而使用独立信息架构：

- 模型版本与训练截止时间。
- 评估区间和预测截点。
- 模型 vs 市场基线的指标差。
- Brier / Log-loss / RPS / ECE。
- 校准曲线与样本置信区间。
- 累计收益、回撤和 CLV。
- 按联赛、赔率区间、置信度、数据模式分层。
- Champion / Challenger 对照。

## 4. 视觉系统

### 4.1 风格

关键词：

- Sports analytics
- Institutional dashboard
- Editorial clarity
- Restrained dark mode
- Data-first

避免：

- 霓虹赛博朋克。
- 大面积发光渐变。
- 泛 AI Lab 口号。
- 用绿色装饰所有可点击元素。
- 大量胶囊徽标。

### 4.2 色彩

| Token | 色值 | 用途 |
|---|---|---|
| `bg` | `#080D14` | 页面背景 |
| `surface-1` | `#0E1622` | 主表面 |
| `surface-2` | `#131E2D` | 次级表面 |
| `border` | `#243247` | 边界 |
| `text` | `#F3F6FA` | 主文字 |
| `muted` | `#A9B4C3` | 辅助文字 |
| `primary` | `#4CC9F0` | 交互与选中 |
| `positive` | `#38D996` | 已验证正优势 |
| `warning` | `#F5B942` | 降级与不确定 |
| `danger` | `#FF6B6B` | 过期、异常、风险 |

规则：

- 主胜/平/客不使用“好/坏”颜色，采用中性且可区分的三色。
- 正收益绿色只用于通过策略门槛的独立价值。
- 所有状态同时提供文字或图标，不只依赖颜色。
- 普通文字对比度至少 4.5:1。

### 4.3 字体

为了保持离线和包体可控，优先系统字体：

```css
font-family:
  Inter,
  ui-sans-serif,
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  "PingFang SC",
  "Microsoft YaHei",
  sans-serif;
```

数字使用：

```css
font-family:
  "SFMono-Regular",
  "Roboto Mono",
  ui-monospace,
  monospace;
font-variant-numeric: tabular-nums;
```

字号下限：

- 桌面正文：14px。
- 移动端正文：16px。
- 辅助标签：不低于 12px。
- 不再使用当前大量 8–10px 正文式文本。

### 4.4 间距与圆角

- 采用 4px 基础网格。
- 页面段落间距：24/32px。
- 表格行高：56–68px。
- 卡片圆角：10–12px。
- 胶囊仅用于真正的状态标签和筛选器。
- 阴影只用于浮层，不用于每个普通卡片。

## 5. 核心组件

### 5.1 Model Status Bar

展示：

- 模型版本。
- 训练截止时间。
- 校准状态。
- 预测截点。
- 数据更新时间。
- 降级数量。

状态异常时固定可见，不藏在页尾。

### 5.2 Match Table

桌面端以表格为主，支持：

- 时间和联赛分组。
- 多列排序。
- 键盘上下移动。
- 固定表头。
- 行内展开。
- 只看价值、弃权、数据异常。

行级必须直接展示：

- 概率。
- 公平赔率。
- 竞彩 SP。
- Edge。
- 置信/弃权状态。
- 数据模式。

### 5.3 Decision State

统一为四种状态：

| 状态 | 含义 |
|---|---|
| `candidate` | 独立数据与门槛通过，进入候选 |
| `lean` | 有方向，但没有足够价格优势 |
| `no_edge` | 概率与价格无优势 |
| `abstain` | 数据、校准或分布外风险不足以输出 |

页面不使用“必选”“稳”“强推”等语言。

### 5.4 Probability Comparison

优先使用对齐表格或分组条形图，不只使用单条堆叠概率：

| 来源 | 主胜 | 平 | 客胜 |
|---|---:|---:|---:|
| 统计模型 | 42% | 29% | 29% |
| 参考市场 | 39% | 31% | 30% |
| 最终校准 | 41% | 30% | 29% |
| 竞彩去水 | 37% | 31% | 32% |

同时显示公平赔率与差值，降低心算成本。

### 5.5 Data Quality

不再用五个等权小卡片表达数据覆盖，改为：

- 总体数据模式：`完整 / 部分 / 市场基线 / 弃权`。
- 展开后查看各层来源和新鲜度。
- 过期数据直接显示年龄，例如“赔率快照已 47 分钟未更新”。

## 6. 图表规范

### 6.1 概率与校准

- 可靠性曲线必须有理想对角线。
- 显示每个桶的样本量。
- 有条件时显示 Bootstrap 置信区间。
- 模型与市场基线同时出现。

### 6.2 收益与风险

- 以初始资金标准化为百分比。
- 同时显示累计收益和回撤。
- 标出下注数量和策略版本变化。
- 不用没有坐标与单位的装饰性折线。

### 6.3 比分矩阵

- 使用可读热力色阶。
- 显示行列标题。
- 可切换概率/公平赔率。
- 移动端只展示 Top 比分与紧凑矩阵入口。

### 6.4 赔率轨迹

- 使用时间横轴。
- 分开显示竞彩与参考市场。
- 标注预测截点。
- 不把开赛后或收盘后数据画入赛前分析区。

## 7. 响应式规则

### 7.1 桌面 ≥ 1024px

- 赛单表 + 右侧详情抽屉。
- 支持高密度扫描。
- 详情抽屉宽 440–520px。

### 7.2 平板 768–1023px

- 赛单表减少次要列。
- 详情改为行内展开。
- 筛选器允许两行，但不横向滚动。

### 7.3 手机 < 768px

桌面表格转换为紧凑比赛卡：

```text
19:30 · 英超 · 数据完整
Arsenal      43%   公平 2.33
Chelsea      28%   竞彩 2.52
平局         29%   Edge +3.1%
候选价值 · 中置信
```

要求：

- 主流程不出现横向滚动。
- 点击区域至少 44×44px。
- 筛选器使用底部抽屉。
- 图表提供数据表替代。

## 8. 可访问性

- 所有交互有清晰 `:focus-visible`。
- 页签、抽屉、排序、筛选均支持键盘。
- `details/summary` 或自定义 disclosure 保持正确语义。
- 图表提供 `aria-label` 和可读数据表。
- 尊重 `prefers-reduced-motion`。
- 打印模式保持黑白可读，不依赖背景色。

## 9. 报告源码结构

最终产物继续是单 HTML，但源码不应继续把 CSS、模板和 JS 压在一个 350 行文件中。

建议：

```text
reporting/
├── templates/
│   ├── report.html
│   ├── partials/
│   │   ├── status_bar.html
│   │   ├── match_table.html
│   │   ├── match_detail.html
│   │   └── backtest.html
├── static/
│   ├── report.css
│   └── report.js
├── view_models.py
└── renderer.py
```

渲染时把 CSS/JS 内联进最终 HTML，保持：

- 无 CDN。
- 无网络请求。
- 单文件。
- 源码可维护和可测试。

## 10. 验收标准

### 视觉与信息

- 1280×900 首屏能看到状态、筛选和至少 6 场赛单。
- 主要数据字号不低于 12px。
- 不使用装饰性大 Hero。
- “弃权/无优势”与“候选价值”同等清晰。
- 模型、参考市场、竞彩 SP 不混在一个概率概念里。

### 响应式

- 375、768、1024、1440px 无页面级横向溢出。
- 手机主流程无内部横向滚动。
- 触控目标不小于 44px。

### 可访问性

- 文本对比度达到 WCAG AA。
- 全流程键盘可操作。
- 图表具有文本替代。
- 无颜色唯一编码。

### 工程

- HTML 单文件离线可用。
- JS 控制台 0 error。
- 报告 fixture 覆盖完整、降级、弃权、价值、回测五种状态。
- 增加桌面与移动端视觉回归截图。

## 11. 已落地设计决策

1. 使用高密度研究工作台替代比赛日大 Hero。
2. 桌面端单场详情使用右侧原生 `dialog` 抽屉；移动端自适应为全宽详情层。
3. 支持按决策状态、Edge、时间和置信度排序，不强制把候选置顶，避免界面暗示下注。
4. 保留必要产品名，减少装饰性英文和全大写标签。
5. 每日报告与回测共享视觉基础，但回测使用独立指标、校准与权益区块。

## 12. 验证记录

- 源码已拆分为视图模型、局部模板、CSS 和 JavaScript，最终仍内联为单 HTML。
- 静态验证覆盖报告标识、离线资源、详情抽屉、筛选、排序、键盘导航、移动端卡片与回测区块。
- Chrome 真实渲染下，1280×900 首屏完整显示状态、筛选器与全部演示赛单，不再由大 Hero 挤压主任务。
- 通过 Chrome DevTools 设备模拟验证 375px 与 320px：`scrollWidth` 分别等于 `375` 和 `320`，主流程无横向溢出。
- 手机赛单使用两列紧凑卡、结论独占一行；触控目标和页签满足 44px 约束。
